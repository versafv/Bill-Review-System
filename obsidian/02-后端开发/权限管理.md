# 权限管理（RBAC）

## 概述

三层权限体系：

| 层级 | 控制内容 | 实现方式 |
|------|----------|----------|
| 菜单权限 | 能看哪些页面 | 前端路由动态注册 + 后端菜单树查询 |
| 操作权限 | 能执行哪些操作 | `@PreAuthorize` + 前端 `v-hasPermi` |
| 数据权限 | 能看哪些数据行 | `@DataScope` 注解动态注入 SQL 条件 |

---

## 核心数据模型

```
sys_user ──N:M── sys_user_role ──── sys_role ──N:M── sys_role_menu ──── sys_menu
```

- `sys_menu.perms` 存储权限标识符，格式为 `模块:实体:操作`（如 `system:user:list`）
- `sys_role.data_scope` 控制数据可见范围：1-全部 2-自定义 3-本部门 4-本部门及子部门 5-仅本人
- admin 用户（userId=1）拥有 `*:*:*` 通配权限，并跳过所有数据范围过滤

## 登录时：权限加载

### 1. 查用户 + 角色（4 表 JOIN）

`SysUserMapper.xml:50-58` (`selectUserVo`)：

```sql
select u.*, r.role_id, r.role_name, r.role_key, r.data_scope
from sys_user u
  left join sys_user_role ur on u.user_id = ur.user_id
  left join sys_role r on r.role_id = ur.role_id
```

### 2. 查角色对应权限字符串

`SysMenuMapper.xml:106-113` (`selectMenuPermsByUserId`)：

```sql
select distinct m.perms
from sys_menu m
  left join sys_role_menu rm on m.menu_id = rm.menu_id
  left join sys_user_role ur on rm.role_id = ur.role_id
where ur.user_id = #{userId}
```

### 3. 构建 LoginUser 并缓存

`SysPermissionService.java:58-88`：
- admin 用户直接给 `permissions = {"*:*:*"}`，roles = `{"admin"}`
- 其他用户遍历角色，逐个查 `sys_menu.perms`，合并到 Set，去重

`TokenService.java:114-125`：
- 生成 UUID token
- `LoginUser`（含 userId、permissions、roles）存 Redis，30 分钟过期
- 返回 JWT（JWT 内只存 UUID，不存用户数据）

### 4. 前端获取

`GET /getInfo`（`SysLoginController.java:72-93`）返回 roles 和 permissions 数组，前端存入 Pinia store。

`GET /getRouters` 调用 `selectMenuTreeByUserId`（**5 表 JOIN**，`SysMenuMapper.xml:77-86`）：

```sql
select distinct m.*
from sys_menu m
  left join sys_role_menu rm on m.menu_id = rm.menu_id
  left join sys_user_role ur on rm.role_id = ur.role_id
  left join sys_role ro on ur.role_id = ro.role_id
  left join sys_user u on ur.user_id = u.user_id
where u.user_id = #{userId}
```

前端将返回的菜单树转为 Vue Router 路由并 `addRoute` 动态注册。

## 后端：操作权限

### 每次请求的认证

`JwtAuthenticationTokenFilter.java:31-43`：
1. 从 Header 取 `Authorization: Bearer <JWT>`
2. 解析 JWT 拿 UUID
3. Redis 查 `LoginUser`
4. 构建 `UsernamePasswordAuthenticationToken` 放入 `SecurityContextHolder`

### @PreAuthorize 方法级鉴权

`PermissionService.java:27-40`，bean 名 `"ss"`：

```java
@PreAuthorize("@ss.hasPermi('system:user:list')")
@GetMapping("/list")
public TableDataInfo list(SysUser user) { ... }
```

`hasPermi(permission)` 逻辑：
1. 从 `SecurityContextHolder` 获取 `LoginUser`
2. 权限字符串存入 `PermissionContextHolder`（供后续 DataScope 复用）
3. 检查 permissions Set 中是否有 `*:*:*` 或匹配的具体权限

支持的方法：
- `hasPermi(permission)` — 单权限
- `hasAnyPermi("permission1,permission2,...")` — 任一
- `hasRole(role)` — 单角色（admin 始终通过）
- `hasAnyRoles("role1,role2,...")` — 任一角色
- `lacksPermi(permission)` — 取反

### SecurityConfig 配置

`SecurityConfig.java:86-118`：
- `@EnableMethodSecurity(prePostEnabled = true)` 开启 `@PreAuthorize`
- CSRF 关闭、Session 无状态
- JWT 过滤器插在 `UsernamePasswordAuthenticationFilter` 之前
- `/login`、`/register`、`/captchaImage` 等放行

## 后端：数据权限

### @DataScope 注解

`DataScopeAspect.java:66-169` 切面在 `@DataScope` 标记的方法执行前，向 SQL 注入 WHERE 片段。

```java
@DataScope(deptAlias = "d", userAlias = "u")
public List<SysUser> selectUserList(SysUser user)
```

### 注入逻辑

遍历当前用户的所有角色，按 `data_scope` 生成对应的 SQL：

| data_scope | SQL 片段 |
|------------|----------|
| 1（全部） | 不加条件 |
| 2（自定义） | `d.dept_id IN (SELECT dept_id FROM sys_role_dept WHERE role_id = ?)` |
| 3（本部门） | `d.dept_id = ?`（当前用户 dept_id） |
| 4（本部门+子部门） | `d.dept_id IN (SELECT dept_id FROM sys_dept WHERE find_in_set(?, ancestors))` |
| 5（仅本人） | `u.user_id = ?` |

生成的 SQL 片段注入到 `BaseEntity.params.dataScope`，Mapper XML 中用 `${}` 拼接：

```xml
<select id="selectUserList" resultMap="SysUserResult">
    select * from sys_user u
    left join sys_dept d on u.dept_id = d.dept_id
    where u.del_flag = '0'
    ${sysUser.params.dataScope}
</select>
```

> `${}` 是原始拼 SQL，`#{}` 是预编译占位。这里使用 `${}` 是因为需要拼整段 SQL 子句。

### 权限字符串匹配

DataScope 还通过 `PermissionContextHolder` 获取当前 `@PreAuthorize` 正在检查的权限字符串，只对**持有该权限**的角色拼接过滤条件。没有该权限的角色会被跳过。

## 前端：权限控制

### v-hasPermi 指令

`src/directive/permission/hasPermi.js`：

```html
<el-button v-hasPermi="['system:user:add']">新增</el-button>
```

从 Pinia `userStore.permissions` 取权限数组。如果数组不含指定权限且没有 `*:*:*`，将该 DOM 元素 `removeChild`。

### v-hasRole 指令

`src/directive/permission/hasRole.js`：同理，检查 `roles` 数组，`"admin"` 角色始终通过。

### 路由级权限

`src/store/modules/permission.js:35-54`：
- `generateRoutes()` — 后端返回的菜单 JSON（`GET /getRouters`）转为 Vue Router 路由对象，动态 `addRoute`
- `filterDynamicRoutes()` — 前端本地定义的路由，用 `auth.hasPermiOr()` / `auth.hasRoleOr()` 过滤

### Pinia 用户 Store

`src/store/modules/user.js`：
- `login()` → `POST /login`，存 token 到 cookie
- `getInfo()` → `GET /getInfo`，存 `roles[]` 和 `permissions[]` 到 state

## 完整请求链路

```
前端路由守卫 → 检查 token → 有则放行
     ↓
前端按钮 v-hasPermi → 检查 permissions → 隐藏/显示
     ↓
HTTP 请求 → JwtAuthenticationTokenFilter
     ↓
Redis 查 LoginUser → 放入 SecurityContextHolder
     ↓
@PreAuthorize → PermissionService.hasPermi()
     ↓ 同时
PermissionContextHolder 存当前权限字符串
     ↓
方法执行 → @DataScope 切面
     ↓
遍历角色 → 匹配权限字符串 → 按 data_scope 拼 SQL
     ↓
注入 params.dataScope → Mapper XML ${params.dataScope}
     ↓
返回数据（已过滤）
```
